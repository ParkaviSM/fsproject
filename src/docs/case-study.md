## Case Study

### Problem

Simple task trackers are everywhere, but most tutorials and starter projects skip the parts that make software actually trustworthy — real authentication, real data persistence, and proper access control. I wanted to build something narrow in scope but genuinely complete: a task tracker that a real person could sign up for, use daily, and trust with their data, rather than a demo that resets on refresh.

### Approach

The app is built with Next.js (App Router) and TypeScript on the frontend, with Supabase handling both authentication and the PostgreSQL database. Rather than trusting client-side logic to keep users' data separate, I enforced that boundary at the database level using Row Level Security policies — every read, insert, update, and delete on the `tasks` table checks `auth.uid() = user_id`, so even if the client were compromised, one user's tasks remain invisible and unreachable to another.

On the interface side, I deliberately avoided default form styling. I designed a calm, editorial visual identity — a serif display font paired with a clean sans-serif body, a warm paper background, and a custom circular checkbox with its own completion animation — so the product had a clear point of view rather than looking templated.

### Result

The live application supports full signup and login, and complete CRUD functionality on tasks — create, mark complete, and delete — with loading, empty, and error states handled explicitly throughout rather than left to blank screens or silent failures. The app is deployed at **https://fsproject-9z4t-iota.vercel.app/signup**, with the source publicly available on GitHub.

If I continued building this, my next priorities would be inline task editing, due-date reminders with sorting, and a properly designed dark mode rather than a simple color inversion.

### What I learned

Building this end-to-end surfaced problems that never show up in tutorials. Twice, my Supabase table was missing its `id` and `created_at` columns because I didn't verify the schema Supabase generated by default — the bug only appeared later as a cryptic "column does not exist" error, which taught me to inspect a table's actual structure immediately after creating it rather than assuming the defaults held. Separately, my first deployment to Vercel failed with an "Invalid supabaseUrl" error, which turned out to be a mismatch between my local `.env.local` file and the environment variables configured in Vercel's dashboard — a good reminder that local success doesn't guarantee a working production build, and that environment configuration deserves the same care as the code itself. Working through both issues gave me a much better instinct for where to look first when something works locally but breaks in production.
